# 评测方法

## 1. 策略矩阵（10 配置）

评测 10 个配置，覆盖项目所有生产中的 RAG 开关组合：

| # | 代号 | `graph_rag_enabled` | `agentic_rag_mode` | `multi_query_enabled` | `graph_rag_query_mode` |
|---|---|---|---|---|---|
| 1 | `off`              | False | `off`               | False | — |
| 2 | `multi_query`      | False | `off`               | True  | — |
| 3 | `agentic_grading`  | False | `grading_only`      | False | — |
| 4 | `agentic_rewrite`  | False | `grading_rewrite`   | False | — |
| 5 | `agentic_full`     | False | `full`              | False | — |
| 6 | `graph_naive`      | True  | `off`               | False | `naive`   |
| 7 | `graph_local`      | True  | `off`               | False | `local`   |
| 8 | `graph_global`     | True  | `off`               | False | `global`  |
| 9 | `graph_hybrid`     | True  | `off`               | False | `hybrid`  |
|10 | `graph_mix`        | True  | `off`               | False | `mix`     |

**互斥关系（已由 chat_service 保证）**：
- Graph RAG 启用时覆盖 Agentic + Multi-Query（最高优先级）
- Agentic grading+ 启用时跳过启发式 `rewrite_query_if_needed`
- Multi-Query 启用时与狭义 Rewrite 互斥

切档实现（`run_rag_bench.apply_config`）：每次切档时先 `PUT .../graph-rag {enabled:false}` 清零，再设 Agentic / Multi-Query，最后按需开 Graph RAG。

## 2. 硬约束

### 全程 `use_web_search=false`

`run_rag_bench.sse_chat()` 在请求 body 里硬编码 `use_web_search: False`。理由：

1. **成本**：SerpAPI 单次 ~$0.015，600 trials × web = $9，且翻墙链路不稳定
2. **语义**：开 web 会污染"Graph RAG 是否真的在 KB 上赢"的判断——web 命中可能"抢走"KB 多跳本该出场的场景

### `use_reranker=true`

所有配置都开 reranker（与项目默认一致），确保策略差异只来自"是否做图 / grading / multi-query"，不引入 reranker 维度噪音。如果要专题测 reranker，另起一个矩阵。

### 同一 seed、同一 KB

- `seed=42` 固定抽样
- KB 一次性入库 118 篇，全 10 配置共用
- Graph RAG 索引**一次构建、所有 graph_* 档位复用**（LightRAG LLM cache 命中率 >95%）

## 3. 指标（4 类）

| 指标 | 类型 | 实现 | 数值范围 | 成本 |
|---|---|---|---|---|
| **Recall@K**（K=4）| 客观 | retrieved refs 的 source（`crud-NNNN.txt`）与 `gold_doc_ids` 求交集 / gold 总数 | [0, 1]，越高越好 | 0 |
| **Faithfulness** | RAGAS | 答案陈述是否来自 retrieved_contexts；LLM judge（DeepSeek-chat）| [0, 1]，越高越好 | ~$0.001 / query |
| **Answer Correctness** | RAGAS | 答案与 `gold_answer` 的语义 + 事实对齐；LLM + embedding | [0, 1]，越高越好 | ~$0.002 / query |
| **平均延迟** | 客观 | SSE 流式请求从 POST 到 `[DONE]` 的 wall-clock | 毫秒 | 0 |

### Recall@K 的 Graph RAG 特殊处理

LightRAG 返回的 context 是 `---Entities--- / ---Relationships--- / ---Sources---` 结构化 markdown。`graph_rag._context_to_documents` 会尝试解析 `---Sources---` 段里的 `file_path` 列，拆成多个 Document 回传；如果解析失败（LightRAG 某些版本输出格式不标准），所有 refs 的 source 会退化成 `graph_rag:<mode>`——这种情况 Recall@K **计为 None 从均值中剔除**（而不是 0），避免"解析失败"和"真没召回"混淆。

### RAGAS 的 judge 与 embedding

`score_rag_bench._build_ragas_llm_and_embeddings()`：

- LLM judge：`ChatOpenAI(model="deepseek-chat", base_url="https://api.deepseek.com/v1")`——**非 reasoner**，便宜 10×
- embedding：`HuggingFaceEmbeddings(model_name=backend/data/models/bge-m3)`——复用项目已下载的本地模型，零额外下载

RAGAS 0.4.x 的 `LangchainLLMWrapper / LangchainEmbeddingsWrapper` 已 deprecated 但未删除，v1.0 前继续可用。

## 4. 数据流

```
 1. prep_crud_eval.py      → queries.jsonl + docs/
       ↓
 2. ingest_eval_kb.py      → POST /api/upload × 118; 可选 POST /api/embedding/graph-rag/build
       ↓
 3. run_rag_bench.py       → for config in 10:
                                  apply_config(config)         # 3 次 PUT 切档
                                  for query in 60:
                                      POST /api/chat/completions (SSE)
                                      解析 content / parsed.refs / [DONE]
                                      写一行到 run_<ts>.jsonl
       ↓
 4. score_rag_bench.py     → 读 jsonl → Recall@K → RAGAS evaluate()
                                  → aggregate by config, category
                                  → report_<ts>.md
```

## 5. 失败处理

- **单条 query 异常**（网络、timeout、后端 500）：`run_rag_bench` 记 `error` 字段但不中断整批，确保同一份 jsonl 里 600 条都在
- **Graph RAG 降级**（`trace.degraded=True`）：后端自动回退到 classical 检索；jsonl 里该条的 refs 会是 classical 的 refs，不体现 graph 能力——**这是特性，用于观测 graph 降级频率**
- **RAGAS LLM judge 挂**：`evaluate(raise_exceptions=False)`，单条失败该指标标 None，不影响其它条
- **Embedding 未激活**：`run_rag_bench` 启动前 preflight 检查 `embedding_ready`，不满足直接退出
- **Graph RAG 索引为空**：只要用到 graph_* 配置就 preflight 检查 `stats.nodes > 0`

## 6. 成本预估（60 query × 10 config = 600 trials）

| 项目 | 量 | 成本 |
|---|---|---|
| Graph RAG 建图 | 118 次 LLM 抽取（DeepSeek-chat，含 cache）| **<$1，一次性** |
| 600 次 chat completions | 每次 1-3 次 LLM，均值 ~2 | **~$2-3** |
| RAGAS 1200 次 judge | faithfulness + correctness × 600 | **~$2-3** |
| **合计** | | **~$5-7，70-90 min** |

首次跑较慢（建图 + embedding 加载），二次复跑 Graph RAG 部分靠 LLM cache 几乎免费。

## 7. 复现性保障

- **seed=42** 固定抽样：同一份 queries.jsonl + docs/ 可以在不同机器上一字不差重建
- **配置档位持久化**：每次切档都落盘 `backend/data/embedding_config.json`，意外断开也能从上次中断处续
- **latest.jsonl 软链**：`results/latest.jsonl` 永远指向最近一次 run，自动化脚本友好
- **build_graph 幂等**：LightRAG 按 doc_id 跳过已入图文档，再跑不重复抽取
